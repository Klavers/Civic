# 작성 예시

핸드오프 메모의 각 섹션을 어떻게 채우는지 보여주는 범용 예시. 프로젝트 도메인에 맞춰 내용만 바꾸면 된다.

---

## 예시: § 1. 세션 요약

```markdown
## § 1. 세션 요약

**기간**: 2026-05-14 ~ 2026-05-16

**최대 성과**:
1. 인증 미들웨어 리팩터링 PR #12 머지 — 세션 토큰 저장 방식 교체
2. 후속 작업(토큰 만료 처리) 사전 조사 완료 — 영향 받는 호출부 8곳 매핑

**머지된 PR**:

| # | 제목 | 머지일 | 내용 |
|-|-|-|-|
| #12 | Refactor session token storage | 2026-05-15 | 미들웨어 3개 파일 수정 |
```

**왜 이렇게 쓰나**: 성과는 명확한 산출물. PR 표는 다음 세션이 `git log`로 바로 매칭 가능.

---

## 예시: § 3. 1순위 작업

```markdown
### 1순위: 토큰 만료 처리 구현 (즉시 착수 가능)

**상태**: 사전 조사 완료. 구현 branch 미생성.

**핵심 사실**:
- 영향 호출부 8곳 (auth/, api/handlers/ 하위)
- 만료 임계값은 설정값 `SESSION_TTL` 단일 소스 — 하드코딩 금지

**PR 구조** (3 커밋):
1. 만료 검사 헬퍼 추가
2. 8개 호출부에 헬퍼 적용
3. 만료 경로 테스트 추가

**테스트 전략**:
- `tests/test_auth.py` — 만료/미만료 케이스
```

**왜 이렇게 쓰나**: 다음 세션이 원본 문서를 다시 읽지 않고도 핵심 사실 + PR 구조 + 테스트 전략을 바로 파악.

## 예시: § 4. 사용자 성향 (이번 세션 고유만)

```markdown
- **단계별 확인 선호**: 이번 세션에서 사용자는 큰 변경을 한 번에 검토하지 않고
  커밋 단위로 끊어 확인 요청. 일괄 dump 반려 경험 있음.
- **권장안 명시 요구**: 옵션 제시 시 매번 "권장: X, 이유: …" 요구.
```

**왜 이렇게 쓰나**: CLAUDE.md에 이미 있는 일반 규범은 링크로만. 이번 세션에서 드러난 특이점만 기재.

---

## 안티패턴 (쓰지 말 것)

1. **모호한 착수 가이드** — "관련 문서 검토 → 구현 착수 → PR 생성" 같은 빈 단계.
2. **CLAUDE.md 규범 복사** — 이미 CLAUDE.md에 있는 협업 규범을 §4에 그대로 옮김.
3. **성과 ≠ 활동 혼동** — "grep 수행", "파일 읽음"은 활동. 성과는 "호출부 8곳 매핑 완료".
4. **향후 우선순위에 "언젠가"** — 구체 계획이 없는 아이디어는 안 씀.
5. **경로 생략** — "plan 파일 읽기" 대신 정확한 경로 명시.
