# 단일 소스 일관성 가이드

여러 plan 문서가 같은 수치·enum·수식을 각자 literal로 재정의하면, 한쪽만 갱신되어 **drift(불일치)** 가 발생한다. 이 가이드는 그 패턴을 감지하고 참조 링크로 치환하는 절차다.

## 원칙

- 어떤 수치·enum·수식·식별자든 **정본(canonical) 문서가 하나**여야 한다.
- 하위·후속 문서는 그 값을 **재정의하지 말고 참조**한다 — "X §부록 Y 참조" 형태.
- 정본 문서가 갱신되면 참조 문서는 자동으로 최신 상태가 된다.

## 단일 소스 등록 (프로젝트별)

프로젝트가 정본 위치를 확정하면 아래 표에 등록하고, 이후 silent 중복 감지의 기준으로 쓴다. (초기 상태에서는 비어 있다.)

| 항목 | 정본 문서 / 위치 | 참조 형식 |
|-|-|-|
| (예: enum 정의) | `plan/PNN_Name_planned.md` §부록 X | "PNN §부록 X 참조" |
| (예: 핵심 수식) | `plan/PNN_Name_planned.md` §M | "PNN §M 참조" |

## 감지 대상 패턴

하위 문서에서 다음이 정본과 중복 재등장하면 silent 중복으로 간주:

- enum/상수 블록 재정의 (`Literal[...]`, `class XxxEnum`, 상수 나열 3개 이상)
- 구체 수치의 literal 재기재 (계수·임계값·범위 등)
- 구체 수식의 재기재 (정본 문서에 이미 정의된 공식)
- 식별자 정의의 재진술

## 감지·치환 알고리즘

정본에 기재된 값이 하위 문서에 **literal로 재등장**하면 다음 순서로 치환 제안:

1. 해당 문단 전체를 "본 항목은 X §YY 단일 소스. 본 문서 내 재정의 금지." 한 줄로 축약
2. 필요한 경우 구체값 1개만 참조 인용 (예: "수식은 PNN §M 의 `<공식>` 을 사용한다.")
3. 변경 전후 diff를 사용자에게 제시하고 확인

**예외**: 사용자가 명시적으로 "단일 소스에 반영하되 하위에도 재기재하라" 요청 시에만 중복 허용.

## 신규 정본 등록 시

새 수치·enum·수식이 처음 정의되면:
1. 어느 문서를 정본으로 둘지 사용자와 확정
2. 위 표에 등록
3. 다른 문서에는 참조 링크만 남김
